# Debian Packaging Patch Mirror

This directory tracks the Debian-specific patch queue for OpenCC so upstream
maintainers can review Debian packaging deltas before a release.

Current baseline:

- Rebased against: upstream `BYVoid/OpenCC` at `1.4.0`
- Original Debian source package baseline: `opencc 1.3.1+ds1-3`
- Patch index: `https://udd.debian.org/patches.cgi?src=opencc`
- Debian packaging source of truth: `https://salsa.debian.org/debian/opencc`

This mirror currently tracks exactly four patches (`0003`, `0004`, `0005`,
`0009`) — that is the complete set; there is no larger series to catch up
on. The non-sequential numbering is simply inherited from Debian's own
quilt series, where patches get dropped over time without the remaining
ones being renumbered; it does not imply that patches `0001`, `0002`,
`0006`, `0007`, or `0008` exist and were left out.

Each mirrored patch was re-diffed against current upstream `1.4.0` sources
so it applies cleanly with `patch -p1 --fuzz=0` (the same strictness
`dpkg-source` uses for the `3.0 (quilt)` format), verified in a scratch
worktree checked out from upstream `master`. Where the surrounding code
had drifted from Debian's baseline (`README.md` badge layout, the
`ENABLE_GTEST` block in `CMakeLists.txt`), the patch context was adjusted
to match; the underlying intent of each patch was preserved and is noted
in the patch header alongside the original
`Forwarded`/`Reviewed-by`/`Last-Update` metadata where known.

Intent:

- Keep Debian packaging deltas visible to upstream maintainers.
- Identify patches that can be replaced by upstream-supported build options
  (three of the four mirrored patches already have a matching upstream
  `USE_SYSTEM_*` CMake option; see `packaging/debian/SPLIT_PROPOSAL.md`).
- Make release-time Debian coordination easier.

This directory is not authoritative Debian packaging. Debian maintainers
should continue to use the Debian Salsa repository and the Debian source
package as the source of truth. A recreation of the Debian packaging flow
(quilt patch application via `dpkg-buildpackage`, validated against a
clean checkout of upstream `master`) lives in the top-level `debian/`
directory alongside this mirror — see `debian/README.source` for how the
two are kept in sync.

Source package handling:

- Upstream's `CMakeLists.txt` already excludes `/packaging/` from CPack
  source archives (`CPACK_SOURCE_IGNORE_FILES`).
- `package.json` uses an npm `files` whitelist, so this directory is not
  packed into the npm package.
- GitHub's autogenerated "Source code" archives include all tracked files
  and cannot be filtered by repository metadata. Use the CPack-generated
  source archive for release assets when this directory must be excluded.
